# 未来最不值钱的，可能就是代码本身

很多程序员现在最焦虑的问题，是这个：

**AI 会不会替我写代码？**

我越来越觉得，这个问题已经不够尖锐了。

真正的问题其实是：

> **当 AI 已经能在几分钟内生成大量可运行代码时，程序员过去最擅长的那部分价值，正在被系统性压价。**

说得更直接一点：

> **未来最不值钱的，可能就是代码本身。**

这句话会让很多人不舒服。
尤其是靠“写代码能力”建立职业安全感的人。

但不舒服，不代表它不对。

我要先说清楚：

我不是在说代码没用了。
我是在说：

> **在 AI coding 已经显著压缩实现成本的前提下，代码正在快速从“核心竞争力”退化成“相对廉价的产物”。**

代码不会消失。
但代码的稀缺性，正在消失。

真正开始暴涨的东西，是另一层：

> **业务定义。**

也就是：

- 这个业务到底是什么
- 哪些规则是核心规则
- 哪些约束绝对不能破
- 哪些案例足以代表真实业务
- 哪些反馈该被升级为规则
- 哪些版本可以进入 UAT
- 哪些变更必须经过 PR 才能进入系统

换句话说：

> **复杂性没有被 AI 消灭。**
> **复杂性只是从实现层，转移到了定义层。**

如果你还在焦虑“AI 会不会写代码”，
那你可能已经焦虑错方向了。

---

## 一、代码这件事，正在快速失去它过去的定价权

过去几十年，软件开发最贵的地方在哪里？

通常在实现层：

- 架构设计
- 模块拆分
- 接口组织
- 状态流转
- 各种 if/else 和边界条件
- 大量手写代码
- 改动和重构的高成本

所以程序员长期形成了一种职业直觉：

> 会写复杂代码 = 高价值。

这个逻辑在过去是成立的。

因为当时实现能力本身就是稀缺资源。

但现在情况在变。

AI 已经在快速吃掉一大块过去非常昂贵的工作：

- CRUD
- 接口拼装
- service / handler / model 模板逻辑
- 各类脚手架代码
- 大量“会写，但很烦”的中层实现
- 甚至相当一部分业务编排代码

更关键的是：

> **AI 不只是能写，它还能改，还能重构，还能在反馈下继续修。**

这意味着什么？

意味着代码实现本身，正在越来越像：

> **一种可扩张、可替换、可反复生成的廉价劳动。**

很多人不愿意承认这一点。
因为这会直接动摇一种长期存在的职业自我定位：

> “我最核心的价值，就是我能把系统写出来。”

很遗憾，这个时代正在变。

未来系统当然还是要被写出来。
但越来越多时候：

> **写出来本身，不再是最稀缺的能力。**

---

## 二、AI 没有让软件变简单，它只是把复杂性搬家了

很多人以为，AI coding 继续进步，软件开发会越来越简单。

这是一种很天真的误解。

现实恰恰相反：

> **软件并没有变简单。**
> **只是复杂性不再主要停留在代码里。**

过去，复杂性主要表现为：

- 写起来复杂
- 改起来复杂
- 重构复杂
- 分支多
- 状态多
- 胶水逻辑多
- 人脑负担大

现在这部分复杂性开始被 AI 吃掉。

于是复杂性“搬家”了。

它搬到了这些地方：

- 业务规则到底是什么
- 哪些规则是稳定规则，哪些只是局部经验
- 哪些案例具有代表性
- 哪些边界情况应该进入定义层
- 哪些系统行为算“对”
- 哪些反馈应该被吸收，哪些只是噪声
- 什么时候系统能进入 preview
- 什么时候才算 release

也就是说：

> **复杂性没有消失，只是从实现层上移到了定义层。**

这件事，很多团队还没意识到。

他们还在用旧时代的眼镜看 AI coding：

- “AI 帮我少写点代码”
- “Copilot 帮我补全”
- “Agent 帮我改 bug”

但真正大的变化不是“写得更快”，
而是：

> **软件生产的重心开始转移。**

---

## 三、低代码不是方向错了，而是它的方法太蠢

说一句会让很多人皱眉的话：

> **低代码并没有看错问题，它只是用了错误的方法。**

低代码真正的洞见其实一直是对的：

> **业务系统需要一层比通用编程语言更贴近业务本身的表达层。**

这件事本身没毛病。

问题出在什么地方？

问题出在传统低代码做了一个几乎注定会失败的假设：

> 平台设计者可以提前预制出足够多的组件、表单、节点、流程、条件、配置项，最后把真实业务塞进去。

这在简单场景当然能跑：

- 审批流
- 简单表单
- 一些固定流程

但一旦业务变复杂，立刻暴露原形：

- 规则不是固定枚举的
- 业务上下文是活的
- 大量例外不是事先能预制的
- 隐性约束和默认判断根本不在配置项里
- 真正重要的业务常识，平台并不知道

最后发生了什么？

> **复杂性没有被消灭，只是被赶到了平台边缘。**

然后出现熟悉的一幕：

- 补脚本
- 写插件
- 手搓扩展
- 到处特判
- 一边说低代码，一边偷偷回到代码

所以我现在越来越认同一个判断：

> **低代码失败，不是因为“业务需要表达层”这个方向错了。**
> **而是因为它试图用预制世界去穷举真实世界。**

这是根本不可能成功的。

---

## 四、真正的业务规则，根本就不是一个纯形式系统

很多工程师有一个隐含信仰：

> 只要规则写得足够多、足够细、DSL 足够强，业务就可以被完整精确定义。

这个信仰在玩具问题里成立，
在复杂业务里经常失效。

为什么？

因为真实业务世界里，到处都是这种东西：

- 上下文依赖
- 优先级判断
- 经验性常识
- 例外覆盖主规则
- 同样的输入，在不同业务阶段含义不同
- 表面一样、内部逻辑完全不同的场景
- 一堆没人写进文档、但所有老员工默认知道的规则

所以更现实的判断是：

> **复杂业务规则，通常无法被一次性、完备、无歧义地写成纯语言对象。**

注意，我不是说规则不能写。
我说的是：

> **不能低成本、稳定地、一次性穷尽。**

这是两回事。

你当然可以继续往文档里加：

- 更多约束
- 更多注释
- 更多 if/else 解释
- 更多补充条款

但只要业务足够复杂，你很快就会发现：

- 文档变厚了，理解不一定更清楚
- 规则变多了，歧义不一定更少
- AI 看到的上下文变长了，但真正的业务理解未必更稳

这就是为什么我越来越不相信“纯规则主义”。

---

## 五、只靠规则不够，只靠案例也不够

如果规则不够，是不是只要给 AI 足够多案例就行？

也不行。

因为“案例主义”同样有天然问题：

- AI 可能只是在拟合表面模式
- 案例覆盖永远不可能完整
- 案例之间可能存在隐性冲突
- 没有硬边界时，AI 很容易“看起来会了，其实没懂”

所以我越来越认同一种更现实的方法：

> **复杂业务必须通过三件东西共同逼近：**
> **Core Rule + Representative Cases + Structured Feedback**

### 1. Core Rule
这是硬骨架。

例如：

- reservation 不能直接扣减实际库存
- confirmed stock count 会成为 next baseline
- normalized quantity 才能进入 inventory-affecting actions
- 某些状态转换本身就是非法的

这些是**不能靠 AI 猜的**。

### 2. Representative Cases
这是现实切片。

例如：

- 按筐入库，按斤销售
- reservation 成功后取消
- 自动盘点与人工盘点不一致
- 标准价和成交价不同，还要抹零

案例的作用不是做附录，
而是告诉 AI：

> **规则在真实世界里到底长什么样。**

### 3. Structured Feedback
这是收敛机制。

例如：

- 系统行为不符合业务预期
- 某个案例过不了
- 某条规则和现有定义冲突
- preview 能点，但逻辑明显不对
- 某个 release 版本在真实试用中暴露漏洞

这些不是普通意义上的“修 bug”。

它们真正的价值在于：

> **暴露定义哪里还不够清楚。**

所以我现在更愿意把定义系统理解成：

> **规则提供骨架，案例提供语义样本，反馈推动定义修正。**

## 六、未来真正需要的，不是下一代低代码，而是“业务定义系统”

这也是我最近最强的一个判断：

> **我们真正需要的，不是更华丽的低代码。**
> **我们真正需要的，是业务定义系统。**

它至少要有三件东西。

### 1. 原始语义（Primitive Semantics）
不是预制完所有业务概念，而是提供最少、最稳、最可组合的原始表达层。

比如：

- Object
- Relation
- Identity
- Action
- Workflow
- Permission
- Rule / Invariant
- Scenario
- Validation Case

这些不是业务本身。
它们是：

> **表达业务核心规则的最小语义单位。**

它们的目标不是完美穷尽业务，
而是让复杂业务能够被**相对精确地表达出来**。

注意，是**相对精确**。
不是完美形式化。

### 2. 演化模型（Draft / Preview / Release）
系统不是一次做完的，而是不断逼近的。

我现在越来越认同一个非常朴素，但非常真实的划分：

#### Draft
有产物了，可以继续和 AI 聊，系统还不能真正跑。

#### Preview
系统可以跑、可以看、可以点，但不好用，主要用于对齐和补案例。

#### Release
进入 UAT / 内测阶段，90% 常见情况已经没什么大问题，可以在受控范围内使用。

更重要的是，它不是线性的，而是循环的：

> **Draft → Preview → Release → 再从某个 Release 切新的 Draft，继续演化。**

这才像真实业务成长。

### 3. 信任机制（Session / Branch / PR）
这一点我认为极其关键，而且以后会越来越重要。

我现在越来越认同一种机制：

> **一个有意义的 session，对应一个 Git 分支。**
> **这个 session 的上下文来源，由它的 base branch 决定。**
> **会话里产生了真正有价值的内容，再通过 PR 进入正式系统。**

也就是说：

- 对话用于探索
- 分支用于承载上下文
- PR 用于正式纳入、审阅、追踪、回滚

这意味着什么？

意味着：

> **Git 未来不只是管理代码，它会越来越多地管理业务认知的演化。**

很多人还没意识到这一点。
但我越来越觉得，它会是未来软件系统信任治理的一部分。

---

## 七、为什么我说“代码正在贬值，业务定义正在暴涨”

因为我越来越确信，未来的软件竞争力，不在这些地方：

- 谁写代码更快
- 谁背的框架更多
- 谁更会拼 API
- 谁更会堆 Agent 工具链

而在这些地方：

- 谁能更快识别核心规则
- 谁能更清楚地组织规则与案例
- 谁能更稳地把复杂业务表达成相对精确的定义
- 谁能让 AI 在定义之上持续生成、验证、修正系统
- 谁能构建可追踪、可回滚、可审阅的定义演化机制

所以我才会说：

> **代码正在贬值，业务定义正在暴涨。**

不是因为代码没用了。
而是因为：

> **在 AI 时代，代码越来越像体力活，定义越来越像智力活。**

---

## 八、最终结论

如果要把这篇文章压成一句话：

> **未来的软件开发核心，不再是实现系统，而是定义系统。**

而这个“定义系统”，不是文档，不是低代码平台，也不是 prompt。

它是：

- 原始语义
- 核心规则
- 代表性案例
- 结构化反馈
- 分阶段演化
- Git / PR 信任机制

共同组成的一套方法论。

所以我最后的判断是：

> **我们不是在做传统低代码，也不是在单纯让 AI 写代码。**
> **我们正在逼近一种新的软件生产方式：**
> 
> - 用一组原始语义相对精确地表达业务核心规则  
> - 用案例不断逼近真实业务  
> - 用验证与反馈持续修正定义  
> - 用 Draft / Preview / Release 管理系统成熟度  
> - 用 Session / Branch / PR 建立可追踪、可回滚、可审阅的信任机制  

这不一定是最后答案。
但我越来越相信，它是一个值得认真讨论的方向。

被骂也值。

因为如果这个判断是对的，
那很多人现在还在焦虑“AI 会不会替我写代码”，
其实已经焦虑错了方向。
